Memory-mapped address mess

This diagram should help to clarify the difference between addressing from Vivado and ARM:

flowchart LR subgraph "Programmable logic" A["`*Zynq 7 Master AXI* (Vivado)`"] subgraph sub1 ["IP core 1"] B["`IP Core 1 (*Master Base Offset* = 0x4040_0000)`"] BA[["`Register 1.1 (*Offset* = 0x00-0x03)`"]] BB[["`Register 1.2 (*Offset* = 0x04-0x07)`"]] BC[["`Register 1.3 (*Offset* = 0x08-0x0B)`"]] end subgraph sub2 ["IP core 2"] C["`IP Core 2 (*Master Base Offset* = 0x4100_0000)`"] CA[["`Register 2.1 (*Offset* = 0x00-0x03)`"]] CB[["`Register 2.2 (*Offset* = 0x04-0x08)`"]] end subgraph sub3 ["IP core 3"] D["`IP Core 3 (*Master Base Offset* = 0x4C00_0000)`"] DA[["`Register 3.1 (*Offset* = 0x00-0x03)`"]] DB[["`Register 3.2 (*Offset* = 0x100-0x103)`"]] DC[["`Register 3.3 (*Offset* = 0x104-0x107)`"]] end end subgraph "TCP server" M["` *int dh = open(#quot;/dev/mem#quot;)* (ARM processor)`"] N["` *unsigned int #ast;ip_core_1 = mmap(...,0x4040_0000)* `"] O["` *unsigned int #ast;ip_core_2 = mmap(...,0x4100_0000)* `"] P["` *unsigned int #ast;ip_core_3 = mmap(...,0x4C00_0000)* `"] end A<-->B A<-->C A<-->D B<-->BA B<-->BB B<-->BC C<-->CA C<-->CB D<-->DA D<-->DB D<-->DC N<-->M O<-->M P<-->M sub2<-->O BA<--> |"`#ast;(ip_core_1 + (0x00>>2))`"| N BB<--> |"`#ast;(ip_core_1 + (0x04>>2))`"| N BC<--> |"`#ast;(ip_core_1 + (0x08>>2))`"| N DA<--> |"`#ast;(ip_core_3 + (0x00>>2))`"| P DC<--> |"`#ast;(ip_core_3 + (0x104>>2))`"| P

Every address you see in Vivado corresponds to single byte (8-bits).

When you map address space of certain AXI interface in Linux using code below, you have to provide the same offset you see in Vivado address editor (i.e. Master Base Offset 0x4040000 by default for DMA) 

#define DMA_AXI_ADDR 0x40400000

// Open /dev/mem which represents the whole physical memory
int dh = open("/dev/mem", O_RDWR | O_SYNC); 

// Create reference (pointer) to mapped memory at offset DMA_AXI_ADDR
// with length 0xFFFF (in bytes)
unsigned int *dma_address = mmap(NULL, 0xFFFF,
                                     PROT_READ | PROT_WRITE,
                                     MAP_SHARED, dh, DMA_AXI_ADDR);

If you afterwards want to access register within the memory space dma_address, you have to bitshift the register offset by 2 bits (or divide by 4), because ARM is addressing at 32-bit increments, such as: 

#define REGISTER_OFFSET 0x100

int new_value_1 = 42;
int new_value_2 = 911;

// This writes new_value_1 to address 0x40400100 and
// new_value_2 to address 0x40400104
*(dma_address + (REGISTER_OFFSET>>2)) = new_value_1;
*(dma_address + (REGISTER_OFFSET>>2) + 1) = new_value_2;

If you want to access specific byte within address, you need to utilize bit-masking: 

uint8_t new_byte = 255;

// This writes new_byte into address 0x40400101
uint32_t old_value = *(dma_address + (REGISTER_OFFSET>>2));
*(dma_address + (REGISTER_OFFSET>>2)) = (old_value & ~((uint32_t)(0xFF00)) ) | ((uint32_t)(new_byte)<<8);

// (old_value & ~((uint32_t)(0xFF00))) This reads register value and erases
// second (LSB) byte. 
//
// ((uint32_t)(new_byte)<<8) This shifts a new byte to appropriate position
// casts it to 32 bit integer. Or (|) between the brackets writes into the byte
